## 2026-04-27 - 条件单重复挂单问题修复（v2.4）

### 问题现象
用户发现在币安App中看到了13条条件委托单，远超预期的2条（止损+止盈）。策略启动后每60秒重复挂SL/TP。

### 问题根因（3个bug叠加）
1. **`newOrderRespType: 'ACK'` 静默失败**：止损/止盈挂单时API不报错但订单实际未成功
2. **策略重启后已有持仓不挂SL/TP**：`open_position`检测到已有持仓时直接return，从不执行挂SL/TP逻辑
3. **`fetch_open_orders`返回0**：`STOP_MARKET`/`TAKE_PROFIT_MARKET`是algo订单，ccxt的`fetch_open_orders`对其返回0（不计入），导致每60秒SL/TP检查一直认为没挂，重复创建新订单

### 修复内容

**v2.4修复：**
1. `open_position`重构：提取`place_sl_tp()`统一挂单函数，已有持仓也调用此函数挂SL/TP
2. 主循环SL/TP检查：改为对比state中SL/TP价格与计算值（不依赖fetch_open_orders）
3. state保存algoId：追踪SL/TP订单ID便于调试
4. ACK改为FULL：保留之前的修复

### 关键代码
```python
# place_sl_tp统一挂单（给新仓和已有持仓复用）
def place_sl_tp(direction, avg_price, qty):
    # 先取消已有条件单
    open_orders = binance.fetch_open_orders(SYMBOL)
    ...
    # 挂止损止盈
    sl_order = binance.create_order(..., newOrderRespType='FULL')
    tp_order = binance.create_order(..., newOrderRespType='FULL')
    return sl_price, tp1_price, sl_algo_id, tp_algo_id

# 主循环SL/TP检查：基于state价格对比
if has_pos and cycle % 6 == 0:
    expected_sl = round(entry * (1 - STOP_LOSS_PCT), 1)
    expected_tp = round(entry * (1 + TAKE_PROFIT_PCT), 1)
    sl_match = abs(stored_sl - expected_sl) <= 1
    tp_match = abs(stored_tp - expected_tp) <= 1
    if not (sl_match and tp_match):
        open_position(...)  # 重新挂单
```

### 教训总结
- **state文件是本地记录，不可靠**：每次操作前必须以交易所实际状态为准
- **API返回成功不等于订单创建成功**：条件单必须验证algoId
- **`newOrderRespType: 'ACK'` 会静默失败**：对条件单必须用 `'FULL'`
- **ccxt的`fetch_open_orders`对STOP_MARKET/TP返回0**：检查这类订单要用价格对比法

### 2026-04-28 00:24 - v2.6：有持仓也继续开仓（累积加仓模式）

**需求：**
- 有仓位时点位到了也开仓，不等待
- 每个仓位都挂独立的止盈止损
- 即使仓位重叠（同一方向多次信号），点位到了也开仓

**改动：**
1. **移除 `has_pos` 限制**：不再检查是否有持仓，有信号就开仓
2. **移除平仓后5分钟冷却**：不再等待，平仓后下个信号立即可以开仓
3. **保留30分钟连亏保护**：连续亏损3次后仍等待30分钟
4. **累积加仓**：已有同向持仓时，追加开仓 + 重设整体 SL/TP
5. **加权平均入场价**：加仓后 SL/TP 按总数量和加权均价重新计算

**关键行为：**
```
旧逻辑：有持仓 → 跳过信号 → 等待平仓冷却 → 再检查信号

```

**注意：**
- 币安 U 本位永续同方向多仓在交易所内合并为一个仓位，但各仓位的 SL/TP 独立存在
- 每个仓位独立计算 SL/TP（按各自的入场价），各仓位的止盈止损互不影响
- 当某一仓位的 SL/TP 触发时，只平该仓位的数量，其他仓位继续运行

### 2026-04-29 13:21 - v2.9.2：修复44条重复SL/TP挂单（价格精度比较bug）

**问题现象：**
- 用户手动撤销SL/TP后，策略在短时间内挂了44条重复挂单
- 交易所实际只有1个持仓，但有44对SL/TP条件单在挂
- 全部是同一个止损价和一个止盈价（SL=79738.4 / TP=73545.1）

**问题根因：**
- state.json 存储的止损价格：`79738.38`（Python float，精确到2位小数）
- 交易所返回的 triggerPrice：`79738.4`（字符串，精确到1位小数）
- SL/TP检查代码：`float(o.get('triggerPrice', 0)) == sl_price`
- `79738.4 != 79738.38`，比较失败 → 每次轮询都判定"SL缺失" → 重复挂单

**修复内容：**
1. **临时修复**：撤销全部44条挂单，重新挂正确的2条
2. **代码修复**：比较时加 `round()` 消除精度差异
   ```python
   # 修复前
   float(o.get('triggerPrice', 0)) == sl_price
   # 修复后
   round(float(o.get('triggerPrice', 0))) == round(sl_price)
   ```
3. **state.json精度修复**：止损止盈价格改为整数（与交易所一致）

**修复文件：**
- `auto_trade.py` 第889、894行

**验证结果：**
- 撤销：44条全部成功撤销
- 重挂：2条正常挂单（SL @ 79738 / TP @ 73545）
- 脚本修复后重启不会再重复挂单

### 2026-05-01 21:08 - v2.9.3：4h/1d均线方向指标赋值错误

**问题现象：**
- 日志一直显示 `1d: 📉 空头`，即使在明显上涨趋势中也显示空头

```python
# 错误代码
ma_dirs['4h'] = 'long' if ma4h > ma4h_prev else 'short'

# 正确代码
ma_dirs['4h'] = 'long' if ma4h > ma4h_prev else 'short'
ma_dirs['1d'] = 'long' if ma1d > ma1d_prev else 'short'   # ✅ 独立计算1d均线方向
```

### 2026-04-29 14:46 - v2.9.2：反向持仓价格间隔检查

**问题现象：**
- $77,415 有空单 → 14:34 又触发做多信号 → 直接开了多单 @ $77,001
- 多空间隔仅 ~$414（0.5%），对冲意义不大

**修复内容：**
有反对方向持仓时，开仓必须与反对方向均价拉开 ≥1.5% 间隔：

```python
# ========== v2.9.2: 有反对方向持仓时，检查价格间隔≥1.5% ==========
opposite = 'short' if sig == 'long' else 'long'
opp_entries = [float(p.get('entryPrice', 0)) for p in actual_positions if p.get('side') == opposite]
if opp_entries:
    opp_avg = sum(opp_entries) / len(opp_entries)
    if sig == 'long':
        # 做多：需低于空单均价1.5%+
        if price >= opp_avg * 0.985:
            log(f"⛔ 做多需低于空单均价1.5%+，跳过")
            continue
    else:
        # 做空：需高于多单均价1.5%+
        if price <= opp_avg * 1.015:
            log(f"⛔ 做空需高于多单均价1.5%+，跳过")
            continue
```

- `auto_trade.py` 第953-968行
